# AI 技术项目执行规范（试行）

## 总则

为提升交付效率、降低协作损耗、强化结果导向，即日起技术项目统一执行以下规范：

一个项目，一个技术主负责人；按项目结果负责，不按前后端割裂负责。

## 1. 技术栈标准

### 1.1 允许技术栈（默认）

后端：

- Go
- PHP

前端/客户端：

- Vue
- React
- Flutter
- iOS 原生
- Android 原生

### 1.2 管控要求

- 新项目默认仅可使用上述技术栈。
- 未经审批，不得引入其他语言、框架或小众方案。
- 技术选型优先级：交付效率 > 维护成本 > 技术新颖性。

## 2. 项目主负责人制度（Owner 制）

### 2.1 责任定义

每个项目必须指定 1 名技术主负责人（Tech Owner），对项目全生命周期负责，包括但不限于：

1. 后端接口
2. 前端页面
3. 数据结构
4. 联调测试
5. 问题修复
6. 上线部署
7. 线上反馈处理
8. 后续维护支持

### 2.2 结果口径

- 不接受“仅前端完成/仅后端完成”作为交付结论。
- 交付标准以“业务可用、流程闭环、上线稳定”为准。

## 3. 方案规范

### 3.1 Web 站点（含 SEO）

涉及收录、首屏性能、内容展示的站点，默认优先：

- Go/PHP 后端 + SSR 前端
- Go/PHP 混编方案

禁止仅按普通前后端分离页面处理 SEO 关键站点。

### 3.2 App 项目

默认优先：

- Go/PHP 后端
- Vue/React/Flutter/iOS/Android

选型建议：

- H5/后台：Vue 或 React
- 跨端 App：Flutter
- 高性能/强原生能力：iOS/Android 原生

## 4. 上线前强制门禁（Go-Live Checklist）

上线前必须全部通过，缺一不可：

1. 功能按需求完成
2. 前后端联调完成
3. 核心流程自测通过
4. PM/运营/测试/业务验收通过
5. 上线时间、版本、影响范围确认
6. 域名/配置/接口/权限检查完成
7. 回滚与应急方案确认
8. 维护责任人确认

未通过门禁项，不得上线。

## 5. 交付移交标准

项目上线后，技术主负责人必须完成以下移交：

1. 代码仓库与分支说明
2. 部署方式与发布流程【后端协助】
3. 服务器/环境信息/资源调度（例如图片、视频）【后端协助】
4. 核心接口说明
5. 关键业务逻辑文档
6. 常见故障处理手册
7. 需求与问题反馈入口

未完成移交，视为交付未完成。

## 6. 立项必备六要素（评审前置）

PM 与技术必须在立项时确认：

1. 项目目标
2. 技术主负责人
3. 技术栈选择
4. 实现方案（前后端）
5. 上线交付标准
6. 后续维护责任人【对应的技术栈负责人】

无上述信息，不得进入开发排期。

## 7. 过程管理要求

技术主负责人需持续履行：

- 周期性同步进度（至少每周）
- 主动暴露风险与阻塞
- 协调跨端一致性
- 保证信息不断层、口径一致

## 8. 例外审批机制（特殊技术栈）

确需引入新技术时，必须提交审批说明：

1. 业务必要性
2. 技术收益
3. 维护成本
4. 团队掌握情况
5. 上线风险
6. 接手成本

未经审批通过，不得使用。

## 9. 追责与生效

- 未按规范执行导致延期、质量事故、无人维护等问题，将按项目责任机制追责。
- 本规范自发布即生效，适用于所有新立项项目。

## 核心原则（最终版）

- 一个项目，一个技术主负责人。
- 按项目结果负责，不按前后端割裂负责，负责全栈。
- 前后端必须联通、口径必须统一。
- 上线必须通过最终交付确认。
- 交付后必须明确维护责任。
- 技术选型服务于交付效率与项目收益。
