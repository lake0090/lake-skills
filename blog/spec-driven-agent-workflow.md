# Spec 驱动的 Agent 工作流

复杂任务不应该直接进入编码。更稳的方式是先把需求变成可检查、可讨论、可交接的 spec。

## 三步流程

1. 确认边界 Requirement
2. 校准技术方案 Design
3. 提供进度清单 Task List

## Requirement

先确认任务边界，而不是直接实现。

需要明确：

- 目标用户是谁
- 成功标准是什么
- 哪些内容在范围内
- 哪些内容明确不做
- 是否有 API、权限、兼容性限制

如果这些信息缺失，Agent 应该先提问，而不是猜测。

## Design

动手前先校准方案。

需要明确：

- 技术栈和现有约定
- 页面结构或模块边界
- 数据流和状态来源
- 可能影响的文件范围
- 验证方式

这一步的目标不是写长文档，而是避免方向性返工。

## Task List

最后把任务拆成可以执行和验收的清单。

每个任务最好包含：

- 目标
- 编辑边界
- 非目标
- 验收标准
- 验证方式
- 风险

## 适用场景

适合：

- 中大型前端页面
- 跨模块重构
- 接口和 UI 同时变化的需求
- 多 Agent 协作任务
- 需求边界不清的任务

不适合：

- 单行修复
- 明确的小 bug
- 已有方案非常清楚的局部改动

## 和 large-task-planner 的关系

`large-task-planner` 就是这个思路的子代理化实现：它不直接编码，而是先产出 Scope、Subtasks、Acceptance Criteria 和 Handoff Prompts。
