# 如何写好 AGENTS.md：让任何 AI 编程 Agent 都能高效理解你的项目


---

## 背景：为什么需要 AGENTS.md

大语言模型（LLM）是无状态函数。**每次会话对 Agent 来说都从零开始**——它对你代码库的唯一了解，就是你输入的 token。

各个工具有自己的"记忆文件"约定：

| 工具 | 默认读取的文件 |
|------|--------------|
| Claude Code | `CLAUDE.md` |
| Cursor | `.cursorrules` / `CURSOR.md` |
| Windsurf | `.windsurfrules` |
| GitHub Copilot | `.github/copilot-instructions.md` |
| Gemini CLI | `GEMINI.md` |
| 通用约定 | `AGENTS.md` |

`AGENTS.md` 是一个逐渐形成的社区通用约定，放在项目根目录，被越来越多的工具识别。如果你同时使用多个 AI 工具，维护一份 `AGENTS.md` 是最省力的做法——各工具的专有文件可以用一行 `@AGENTS.md` 引用它。

---

## 核心认知：指令越多，质量越差

这是最反直觉、也最重要的一点。

LLM 在指令数量超过阈值后，**不是只忽略靠后的指令——而是开始均匀地忽略所有指令**。你往文件里多写一条规则，所有规则都变得更脆弱。

另一个实际约束：Agent 工具本身的系统提示词占用了大量 token，你的 `AGENTS.md` 和它共享同一个上下文窗口。文件越臃肿，实际被遵循的概率越低。

**实用自检标准：**

对文件里的每一行问自己——"删掉这行，Agent 会因此犯错吗？"如果不会，删掉它。

经验上限是 200 行，但更好的测试是：你能在两分钟内背出文件的核心内容吗？做不到说明文件太长了。

---

## 写什么：新工程师测试

判断一条内容是否值得写进 `AGENTS.md`，用这个标准：

> **如果一个新来的高级工程师加入你的项目，哪些事情不明确告诉他，他第一天就会搞错？**

那些就是该写的。

### 该写的

- **项目特有的反直觉约定**：比如"我们用 `pnpm` 不用 `npm`"、"枚举类型用 string union 不用 enum"
- **关键依赖和影响范围**："`shared/` 包有零外部依赖，改动会影响所有其他包，需格外谨慎"
- **禁止清单**：不能做什么，往往比正向指令更有效，因为边界更清晰
- **技术栈版本**：精确到版本号，Agent 无法从代码推断你用的是哪个版本
- **常用但不易推断的命令**：不是所有命令都能从 `package.json` 猜出来

### 不该写的

- `"写整洁的代码"` `"取有意义的变量名"` `"适当加注释"`——Agent 本来就会，这是在浪费 token
- ESLint、Prettier 能自动处理的格式规则——用工具解决，不用文字解决
- 项目当前状态、待办事项——这些会变成"死信息"，过期之后比没有还糟
- 三十条格式规则——Agent 不会全部记住，而且会让它忽略你真正想强调的部分

## 写法：指令要"可验证"，不要"可解释"

这是另一个常被忽视的细节。

**❌ 需要判断的指令：**
```
代码要保持简洁
注意错误处理
```

Agent 不知道你的"简洁"标准是什么，不知道"注意"意味着什么程度。

**✅ 可验证的指令：**
```
单个函数不超过 50 行
所有 async 函数必须用 try-catch 包裹，catch 块必须调用 logger.error()，
返回统一的 { error: string, code: number } 结构
```

后者 Agent 自己就能检查是否达标。禁止清单之所以比正向指令有效，本质也是这个原因——边界比"应该怎样"更容易判断。

---

## 结构：用"指针"代替"内容"

不要把所有规范直接塞进 `AGENTS.md`，而是**告诉 Agent 去哪里找**。Agent 会在需要时主动读取，避免占用宝贵的上下文窗口。

**❌ 臃肿写法：**

```markdown
## API 规范
- 路径用 kebab-case
- 属性用 camelCase
- 列表接口必须支持分页
- URL 包含版本号 /v1/、/v2/
- 错误统一返回 { code, message, data }
- 认证使用 Bearer token
...（继续写 20 行）
```

**✅ 指针写法：**

```markdown
## 更多规范
- API 设计规范 → @docs/api-conventions.md
- Git 工作流 → @docs/git-workflow.md
- 架构决策记录 → @docs/adr/
```

**重要前提：** 被引用的文档本身要准确、及时更新。指针只解决了上下文膨胀问题，不解决文档质量问题。指向一份过时文档，比没有还糟——Agent 会认真读，然后按过时的规范执行。

---

## 分层：AGENTS.md vs 按需文档

`AGENTS.md` 只放**每次会话都需要的全局上下文**。场景化的规范放到单独文档里按需引用。

| 放 AGENTS.md | 放单独文档 |
|-------------|----------|
| 项目用 ES Modules 不用 CommonJS | 审查 API 代码时的详细规范 |
| 包管理器用 pnpm | 写测试时的断言风格指南 |
| 改 shared/ 会影响所有包 | 数据库迁移操作流程 |
| 禁止直接修改 migrations/ 下的已有文件 | 部署前的检查清单 |

这个分层对应 Claude Code 的 `SKILL.md`、Cursor 的 Rules 层级等——各工具叫法不同，思路一致：**全局 vs 按需**。

## 实用模板

```markdown
# [项目名]

[一句话描述这是什么项目，让 Agent 快速定位项目性质]

## 技术栈

- 语言/框架：[名称 + 精确版本号]
- 包管理器：pnpm（不用 npm 或 yarn）
- 测试框架：[名称]
- 数据库：[名称 + 版本]

## 常用命令

```bash
pnpm dev          # 启动开发服务器
pnpm test         # 运行测试
pnpm lint         # 代码检查
```

## 架构约束 IMPORTANT

- **禁止**直接修改 `migrations/` 下的已有文件，只能新增迁移
- **禁止**在路由层直接写 SQL，必须通过 Service 层
- 改动 `shared/` 包会影响所有其他包，需格外谨慎
- `legacy/` 目录下的代码禁止修改，只能读取

## 项目特有约定

[新工程师第一天不说就会搞错的事，每条用一句话说清楚]

## 更多信息

- 架构设计 → @docs/architecture.md
- API 规范 → @docs/api-conventions.md
- Git 工作流 → @docs/git-workflow.md
```

## 维护：像代码一样对待它

**检入 git，团队共同维护。** `AGENTS.md` 的价值随时间积累——每次踩坑之后更新它，它会越来越准确。

**出问题先看这里。** 如果 Agent 持续犯某个错误，先检查 `AGENTS.md` 里有没有相关规则，措辞是否清晰，文件是否已经太长导致规则被忽略。

**用强调词标记硬性约束。** 真正不能违反的规则加上 `IMPORTANT` 或 `YOU MUST`，但要节制使用——用词强度要和规则重要性匹配，全部都是 `IMPORTANT` 等于没有 `IMPORTANT`。

**让 Agent 自己来审查。** 项目稳定后，可以让 Agent 读取当前 `AGENTS.md`，问它："哪些规则在实际工作中从来没被用到？" 它的反馈往往比人的直觉更准确——能告诉你哪些规则在上下文里实际上是透明的（被忽略了）。

**定期修剪。** 每个月花五分钟读一遍，删掉已经不再适用的规则。写完之后删掉一半，效果可能反而更好。

---

## 一句话总结

`AGENTS.md` 的核心是**克制**。它不是把所有规范倾倒进去的地方，而是帮 Agent 快速对齐"你项目里反直觉部分"的最小必要上下文。

质量标准只有一个：让一个对你项目一无所知的 AI，读完这份文件，能在第一天就做出符合你期望的决策。
